# StyleSlice 收藏语义搜索 MVP 设计

## 1. 目标

用户可以输入自己仍然记得的颜色、材质、构图或整体感觉，例如“蓝色的、很安静、留白很多”，从自己的默认收藏夹中找回过去保存的截图。

MVP 只实现两路召回：

1. 语义向量搜索，理解近义表达和感觉描述；
2. 简单关键词搜索，保证明确的颜色名、材质名和收藏名称能够命中。

成功标准是：在一组人工标注的真实查询中，目标收藏进入前 5 条结果的比例不低于 80%。

## 2. 前置条件

搜索功能依赖匿名 Supabase Auth 和 RLS 已经完成：

- 每条 `styles` 记录必须包含 `user_id = auth.uid()`；
- 查询只能在当前登录用户的数据范围内执行；
- 服务端不得信任客户端传入的 `user_id`；
- 搜索 RPC 使用调用者会话并遵守 RLS，不能用 `service_role` 直接返回搜索结果。

## 3. MVP 范围

### 3.1 包含

- 一个自然语言搜索输入框；
- 基于现有风格分析生成内部 `search_text`；
- 收藏保存或分析更新时生成文本向量；
- 语义相似度和简单关键词命中合并排序；
- 展示原始截图、收藏名称和已有关键词；
- 向量服务不可用时自动降级为关键词搜索；
- 为现有收藏执行一次向量回填。

### 3.2 不包含

- 专业 Lab 色差计算；
- 颜色词自动转换为色值；
- 错别字纠正和拼音搜索；
- 搜索历史、热门搜索和多轮追问；
- 用户手写“为什么收藏”备注；
- AI 生成匹配原因；
- 复杂筛选器和搜索结果分页；
- 为小规模个人收藏提前进行向量索引调优。

## 4. 可搜索内容

每份收藏生成一段固定顺序的内部文本。它不直接展示给用户，也不包含 Typography：

```text
名称：{name}
概述：{summary}
关键词：{keywords}
颜色：{color names and hex values}
构图：{layout rules}
形状：{shape rules}
图像：{imagery rules}
效果与材质：{effect rules}
视觉组件：{component rules}
必须保持：{mustKeep rules}
明确避免：{avoid rules}
```

`buildSearchText(analysis)` 必须是纯函数。相同分析结果始终生成相同文本，方便测试、重新索引和模型迁移。

## 5. 数据结构

在启用 `vector` 扩展后，为 `public.styles` 增加：

```sql
search_text text
search_embedding extensions.vector(1024)
embedding_model text
embedding_status text not null default 'pending'
search_updated_at timestamptz
```

`embedding_status` 只允许：

- `pending`：尚未生成；
- `ready`：向量可用于搜索；
- `failed`：生成失败，仍可参与关键词搜索。

向量维度固定为 1024。MVP 沿用现有火山方舟供应商，使用支持中文的豆包向量模型；模型接入点通过服务端环境变量 `ARK_EMBEDDING_MODEL` 配置。`embedding_model` 保存实际模型标识，禁止混合比较不同模型生成的向量。

个人收藏规模较小时允许顺序扫描。达到性能阈值后再增加 HNSW 索引，不把索引调优作为 MVP 阻塞项。

## 6. 建立索引流程

收藏首次保存或分析内容更新后：

1. 调用 `buildSearchText`；
2. 保存 `search_text`，将状态改为 `pending`；
3. 服务端调用向量模型生成 1024 维向量；
4. 成功后保存向量、模型标识、更新时间，并改为 `ready`；
5. 失败后改为 `failed`，保留收藏和 `search_text`。

生成向量失败不得导致收藏保存失败。MVP 提供一次重试；重试仍失败时，等待下一次分析更新或后台回填。

已有收藏通过一次性回填脚本分批处理。回填只更新 `pending`、`failed` 或模型标识已经过期的记录。

## 7. 查询流程

前端调用：

```http
POST /api/search
Content-Type: application/json

{
  "query": "蓝色的、很安静、留白很多",
  "limit": 20
}
```

接口规则：

- 必须存在有效 Supabase 会话；
- `query` 去除首尾空白后长度为 1–200 个字符；
- `limit` 默认 20，最大 30；
- 不接受 `user_id`；
- 不记录原始查询文本到持久化日志。

服务端执行：

1. 将查询按空白和常见标点拆成简单关键词；
2. 调用与收藏相同的向量模型生成查询向量；
3. 调用 `search_own_styles` RPC；
4. RPC 在当前 RLS 会话中分别计算语义得分和关键词得分；
5. 合并、排序并返回当前用户的收藏。

初始排序公式：

```text
final_score = semantic_score * 0.8 + keyword_score * 0.2
```

- `semantic_score` 为余弦相似度，范围归一化到 0–1；
- `keyword_score` 为查询关键词在 `search_text` 中的命中比例；
- 无向量的收藏将 `semantic_score` 视为 0，但仍可通过关键词进入结果；
- 不加入收藏时间权重。

向量模型调用失败时，接口直接执行关键词查询并返回结果，不将整个搜索判定为失败。

## 8. 返回结构

```json
{
  "query": "蓝色的、很安静、留白很多",
  "mode": "hybrid",
  "results": [
    {
      "styleId": "style_xxx",
      "name": "银色安静未来感",
      "coverImageId": "image_xxx",
      "keywords": ["雾蓝", "安静", "留白"],
      "score": 0.87
    }
  ]
}
```

`mode` 取值：

- `hybrid`：语义和关键词均可用；
- `keyword_fallback`：向量服务失败，仅使用关键词。

前端不展示原始分数，只用于排序和调试。

## 9. 移动端交互

默认收藏夹顶部提供搜索入口。进入搜索后自动聚焦输入框，提示文案为：

> 还记得什么？试试颜色、材质、感觉或画面……

输入停止 350 毫秒后发起搜索；少于 1 个有效字符不请求。新请求发出时取消旧请求，避免较慢的旧结果覆盖新结果。

结果使用单列截图卡片，展示：

- 代表截图；
- 收藏名称；
- 最多 3 个已有关键词。

状态设计：

- 初始：显示 3 条搜索例句；
- 搜索中：显示轻量加载状态，保留旧结果；
- 无结果：建议换用颜色、材质或整体感觉描述；
- 降级：正常展示关键词结果，不弹出阻塞错误；
- 会话失效：重新创建匿名会话后允许重试。

## 10. 安全与隐私

- API 从用户 JWT 建立 Supabase 客户端；
- RPC 使用 `security invoker` 并受 `styles` 表 RLS 约束；
- 不使用 `service_role` 执行面向用户的查询；
- 返回结果前不得依赖客户端过滤 `user_id`；
- 日志可以记录耗时、结果数量和错误类型，但不记录查询原文、截图内容或完整分析文本；
- 查询接口实施基于会话和 IP 的基础速率限制。

## 11. 测试与验收

### 11.1 单元测试

- `buildSearchText` 包含规定字段且永久排除 Typography；
- 查询清洗和关键词拆分结果稳定；
- 排序公式在语义、关键词和缺少向量三种情况下正确；
- 查询长度和 `limit` 校验正确。

### 11.2 集成测试

- 用户 A 无法搜索到用户 B 的收藏；
- 明确颜色词可以通过关键词命中；
- 感觉描述可以通过语义搜索命中；
- 向量服务失败时仍返回关键词结果；
- 新建和更新收藏会重建 `search_text` 和向量；
- 删除收藏后不再出现在搜索结果中。

### 11.3 质量验收

准备至少 30 份具有差异的测试收藏，以及至少 20 条自然语言查询。每条查询预先标注 1–3 个预期收藏。验收要求：

- 目标收藏进入前 5 条的查询比例不低于 80%；
- 明确名称和颜色查询的目标收藏进入前 3 条；
- 跨用户数据泄漏为 0；
- 向量服务失败时关键词搜索仍可用。

## 12. 后续演进

只有真实搜索日志和用户反馈证明需要时，再按顺序考虑：

1. 用户收藏备注；
2. `pg_trgm` 错别字和近似文字匹配；
3. Lab 色彩距离；
4. 更复杂的混合排序与匹配原因；
5. HNSW 索引和异步向量任务队列。
