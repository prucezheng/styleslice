# StyleSlice｜风格切片 PRD v1.1（优化迭代）

> 基于 v1.0 的用户反馈与测试问题，对三个关键体验问题进行修复性迭代
> 文档版本：v1.1
> 日期：2026-07-23
> 前置文档：StyleSlice_风格切片_PRD_v1.0.md

---

## 1. 迭代背景

v1.0 MVP 完成了「上传参考图 → AI 风格拆解 → 生成 MD + Style Card → 保存资料库」的核心闭环。内部测试和早期用户反馈暴露了三个阻碍产品可用性的问题：

| # | 问题 | 严重程度 | 影响 |
|---|------|----------|------|
| 1 | 产出结果的文字是对照片的解析描述，不是用户可直接使用的提示词 | **P0 阻塞** | 用户拿到结果后无法直接复制到 Midjourney / DALL·E / 豆包等 AI 工具使用，核心价值链路断裂 |
| 2 | 允许输入多张图片增加了分析失败率和风格冲突概率 | **P0 简化** | MVP 阶段多图分析的跨图片归纳不稳定，且增加了用户决策负担 |
| 3 | 解析失败后停留在失败页面，不清空已上传图片 | **P0 体验** | 用户需要手动刷新页面或重新选择文件，增加挫败感 |

本次迭代**只解决这三个问题**，不涉及其他功能变更。

---

## 2. 问题 #1：产出结果从「照片解析」改为「用户可用的 AI 提示词」

### 2.1 现状问题

当前 `renderPrompt()` 的输出偏重对原有图片的分析性描述，例如：

- "使用以下配色方案：暖白背景（#F5F1EA）、深褐主色（#3C2A1A）..."
- "构图方面：画面信息密度较低，留白方面大面积留白..."

这类文字本质上是**对参考图片的逆向工程报告**，而非用户可以直接复制到 AI 生图工具中使用的正向提示词。用户的核心诉求是：

> "我上传了一张我喜欢的参考图，给我一份提示词，让我能在其他 AI 工具里生成类似风格的新图片。"

### 2.2 目标

**将产出结果从「照片解析报告」转型为「正向 AI 生图提示词」。**

### 2.3 功能需求

#### 2.3.1 提示词生成策略

提示词必须满足以下标准：

1. **自然语言正向描述**：用「将要生成什么」代替「图片里有什么」。不说"画面中使用了圆角"，而说"使用大圆角卡片"。
2. **可直接复制使用**：输出的提示词不加 markdown 标记，是纯自然语言段落，用户选中即复制、粘贴即用。
3. **风格还原优先**：提示词的核心目标是让 AI 生成工具还原参考图的视觉风格，而不是描述参考图的内容。
4. **分层输出**：提供两个版本的提示词——
   - **精简版（默认）**：120 字以内，适合 Midjourney / DALL·E 快速出图
   - **完整版**：包含所有视觉规则，适合需要精确控制的高级用户
5. **内容结构**（精简版）：
   ```
   [风格形容词]风格，[配色方案]，[构图特征]，[形状/形态]，[材质纹理]，[关键约束]
   ```
6. **内容结构**（完整版）：
   ```
   [风格名称与一句话定义]
   [正向视觉描述 - 转为生成指令语气]
   [配色方案 - 含色值供参考]
   [构图、间距、留白规则]
   [形状、边框、圆角规则]
   [材质、纹理、效果规则]
   [图像/插画风格]
   [必须包含的元素]
   [严格避免的元素]
   ```

#### 2.3.2 提示词语气转换规则

AI 分析 JSON 中的字段本质是「描述性」的，需要在 `renderPrompt()` 中做语气转换：

| 分析结果（描述性） | 提示词（指令性） |
|---|---|
| "信息密度较低，大面积留白" | "画面留白充足，元素稀疏排布" |
| "图片中使用了大圆角卡片" | "使用大圆角（12-16px）卡片容器" |
| "暖白背景色" | "使用暖白（#F5F1EA）背景" |
| "没有使用阴影" | "扁平设计，无阴影" |
| "避免高饱和度颜色" | "保持低饱和度配色" |

#### 2.3.3 UI 展示调整

在详情页（DetailScreen）中：

- 默认以**精简版提示词**作为主展示内容（替换当前的 `prompt` 展示区）
- 精简版下方提供「查看完整提示词」展开按钮
- 原有 MD 仍然保留在「完整规范」标签页中（供需要完整视觉语言规范的用户使用）
- 复制按钮默认复制精简版；完整版有自己的复制按钮

#### 2.3.4 验收标准

- [ ] 用户上传一张图片后，详情页展示的提示词以「生成一张…风格的图片」开头
- [ ] 提示词是自然语言段落，不含 markdown 标记
- [ ] 提示词中不出现"图片中可以看出"、"原图使用了"等分析性措辞
- [ ] 精简版提示词 ≤ 120 字，可直接复制到 Midjourney 等工具使用
- [ ] 完整版提示词包含所有视觉规则，适合高级用户
- [ ] 测试：将提示词粘贴到任意 AI 生图工具中，生成的新图能还原参考图的视觉风格特征（颜色、构图、材质），而非还原参考图的内容

## 3. 问题 #2：约束只能输入一张图片

### 3.1 现状问题

当前允许一次上传最多 10 张图片（`multiple` 属性 + `MAX_COUNT = 10`）。MVP 阶段多图片跨图片归纳分析不稳定，容易出现风格冲突、分析失败或结果模糊。

### 3.2 目标

**MVP 阶段严格限制用户一次只能上传/分析一张图片。**

### 3.3 功能需求

#### 3.3.1 前端改动

| 改动点 | 位置 | 说明 |
|--------|------|------|
| 移除 `multiple` 属性 | `page.tsx` 第 338 行 `<input>` | 改为单文件选择 |
| `chooseFiles()` 只取第一张 | `page.tsx` 第 147-155 行 | `accepted.slice(0, 1)` 改为只取 `accepted[0]` |
| 单图展示优化 | `page.tsx` 上传区 | 单张图片使用大图预览而非网格 |
| 文案调整 | `page.tsx` | "Upload Reference Images" → "Upload a Reference Image" |
| 上传后再次点击上传区 | `page.tsx` | 替换当前图片（而非追加） |

#### 3.3.2 后端改动

| 改动点 | 位置 | 说明 |
|--------|------|------|
| `MAX_COUNT` 改为 1 | `app/api/upload/route.ts` | `const MAX_COUNT = 1` |
| 错误提示更新 | `app/api/upload/route.ts` | "一次最多上传 1 张图片" |

#### 3.3.3 API 契约更新

`POST /api/upload`：
- 字段名 `file`（单数）而非 `files`（或同时兼容，但只处理第一个）
- 返回单张图片信息（或保持 `images` 数组但长度始终为 1）

`POST /api/analyze`：
- `imageIds` 数组长度限制为 1

#### 3.3.4 验收标准

- [ ] 文件选择器只能选择单张图片
- [ ] 拖拽多张图片时，只取第一张并提示"当前仅支持单张图片分析"
- [ ] 已有图片时再次上传，替换而非追加
- [ ] 上传 API 拒绝超过 1 张图片的请求（返回 400）
- [ ] 分析 API 拒绝超过 1 个 imageId 的请求（返回 400）

## 4. 问题 #3：解析失败后返回空的上传页面

### 4.1 现状问题

当前代码（`page.tsx` 第 222-224 行）：

```typescript
} catch (err) {
  setStage("idle");
  setMessage(err instanceof Error ? err.message : "分析失败，请重试");
}
```

失败时只将 stage 重置为 idle，显示错误信息，但：
- 已上传的图片仍然保留在页面上
- 用户看到的是一个「有错误提示的旧状态」，而非一个干净的新起点
- 用户需要手动操作才能重新开始

### 4.2 目标

**解析失败后，页面完全回到初始空白上传状态，用户可以无摩擦地重新开始。**

### 4.3 功能需求

#### 4.3.1 失败处理流程

```
上传 → 分析中 → 失败
                  ↓
        1. 停止 loading
        2. 清空 files[]
        3. 清空 previews[]（释放 blob URL）
        4. 清空 uploaded[]
        5. 重置 stage = "idle"
        6. 展示错误 toast（3 秒自动消失）
        7. 页面回到空白上传状态
```

#### 4.3.2 具体改动

| 改动点 | 位置 | 说明 |
|--------|------|------|
| 失败时调用 `resetAll()` 函数 | `page.tsx` catch 块 | 清空 files、previews、uploaded、stage、selectedStyle |
| 错误信息改为 toast 形式 | `page.tsx` | 3 秒自动消失的轻提示，不残留错误文字 |
| Toast 包含失败原因 | `page.tsx` | 简短原因（如"AI 分析超时，请重试"）+ 建议 |
| 上传区完全清空 | `page.tsx` | 回到空白上传区（upload-empty 状态） |

#### 4.3.3 resetAll() 函数定义

```typescript
function resetAll() {
  previews.forEach((url) => URL.revokeObjectURL(url));
  setFiles([]);
  setPreviews([]);
  setUploaded([]);
  setStage("idle");
  setSelectedStyle(null);
}
```

#### 4.3.4 错误 Toast 组件

- 从页面底部滑入
- 显示错误简述（如"风格分析失败：模型响应超时"）
- 3 秒后自动消失
- 红色/橙色警示色调
- 不阻塞用户重新上传

#### 4.3.5 验收标准

- [ ] 模拟 AI 服务不可用时（断网/超时/502），页面回到空白上传状态
- [ ] 错误 toast 在 3 秒内自动消失
- [ ] 错误 toast 消失后，页面上没有任何残留的错误信息
- [ ] 用户可以在 toast 消失前或消失后立即重新上传图片
- [ ] 图片上传阶段的失败同理（如格式不支持），也回到空白状态
- [ ] 不会出现「旧图片还在 + 错误信息」的中间态

## 5. 非功能需求

### 5.1 兼容性

- 本次迭代为增量修改，不改变现有数据结构（StyleResult / StyleAnalysis）
- 不改变资料库存储格式
- 不改变现有 API 的返回结构（仅收紧输入约束）
- 不改变 System Prompt（AI 仍然收到同一份指令，只在 `renderPrompt()` 层做语气转换）

### 5.2 回滚方案

- `renderPrompt()` 函数保留旧版本作为 `renderPromptLegacy()`
- 单图约束可通过配置项 `MAX_IMAGES = 1` 控制，未来恢复多图只需改常量
- 失败重置逻辑不影响成功路径

## 8. 测试用例

### TC-01：单张图片上传
1. 进入首页
2. 点击上传区，选择一张图片 → 图片在预览区显示
3. 再次点击上传区，选择另一张图片 → 旧图片被替换为新图片
4. 拖拽多张图片到上传区 → 仅第一张被接受

### TC-02：提示词质量
1. 上传一张暖色调极简风格的设计参考图
2. 等待分析完成
3. 详情页展示精简版提示词 → 以"生成一张…风格的图片"开头
4. 检查提示词中无"图片中可以看出"、"原图使用了"等分析性措辞
5. 展开完整版提示词 → 包含所有视觉规则
6. 将精简版提示词复制到豆包/Midjourney 测试 → 新图还原了参考图的配色、构图和材质风格

### TC-03：分析失败恢复
1. 配置错误的 API Key
2. 上传图片并点击分析
3. 分析失败 → 页面回到空白上传状态（upload-empty）
4. 底部弹出错误 toast → 3 秒后自动消失
5. 用户可以立即重新上传图片

### TC-04：上传失败恢复
1. 上传一个超大文件（>20MB）
2. 上传失败 → 页面回到空白状态 + toast 提示
